## Problem Statement

Jerry는 여러 프로젝트의 작업을 관리할 도구가 필요하다. Jira는 회사 공용이라 "비밀 무기"로 못 쓰고, 개인용 태스크 관리 CLI가 필요. 나중에 factory-run의 티켓 시스템으로 통합.

## What Makes This Cool

개인 Jira. 터미널에서 tk issue create 치면 티켓 생성. tk board 치면 칸반 보드. factory-run이 tk를 호출해서 자동으로 티켓 생성/상태 변경.

## Constraints

- Bun + TypeScript
- SQLite (Bun 내장, bun:sqlite) + 파일 하이브리드
- 중앙 저장소: ~/.config/jerry-tickets/
- 1인 사용, CLI only, 외부 연동 없음 (v1)
- factory-run 통합은 v2에서

## Tech Stack

- Runtime: Bun 1.x
- Language: TypeScript (strict)
- DB: bun:sqlite (내장, 설치 불필요)
- CLI Framework: 가벼운 파서 (citty 또는 Bun.argv 직접)
- 파일 저장: ~/.config/jerry-tickets/{project}/files/{ticketId}/

## Architecture: 하이브리드 저장소

### SQLite (빠른 검색/필터)

~/.config/jerry-tickets/tickets.db

```sql
CREATE TABLE tickets (
  id TEXT PRIMARY KEY,           -- SDLC-001, APP-002...
  project TEXT NOT NULL,
  title TEXT NOT NULL,
  description TEXT,
  status TEXT DEFAULT 'backlog', -- backlog/running/paused/done/aborted
  priority INTEGER DEFAULT 2,   -- 0(긴급) ~ 3(낮음)
  tags TEXT,                     -- JSON array: ["backend", "api"]
  pipeline TEXT,                 -- small/medium/large (factory-run용)
  estimated_hours REAL,
  started_at TEXT,
  completed_at TEXT,
  paused_at TEXT,
  created_at TEXT NOT NULL,
  updated_at TEXT NOT NULL
);

CREATE TABLE ticket_history (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  ticket_id TEXT NOT NULL,
  event_type TEXT NOT NULL,      -- created/started/paused/resumed/completed/comment/handoff
  data TEXT,                     -- JSON
  created_at TEXT NOT NULL,
  FOREIGN KEY (ticket_id) REFERENCES tickets(id)
);

CREATE TABLE tags (
  name TEXT PRIMARY KEY,
  color TEXT
);

CREATE TABLE schema_version (
  version INTEGER PRIMARY KEY,
  applied_at TEXT NOT NULL
);
```

### 파일 시스템 (큰 콘텐츠)

~/.config/jerry-tickets/{project}/files/{ticketId}/
  handoff.json, research/, prd/, specs/, ...

SQLite = 메타데이터 + 히스토리 (검색/필터)
파일 = 산출물, 큰 문서 (읽기/쓰기)

## CLI Interface (gh, jira-cli, linear-cli 패턴)

리소스 + 액션 패턴: `tk <resource> <action> [args] [flags]`

### issue (i) -- 티켓 관리

```bash
tk issue create "검색 API 자동완성"                  # 생성
tk issue create "결제 버그" -p 0 -t backend,urgent   # 우선순위 + 태그
tk issue create "알림 시스템" --est 4h               # 예상 시간
tk issue list                                        # 현재 프로젝트 목록
tk issue list --status running                       # 상태 필터
tk issue list --tag backend                          # 태그 필터
tk issue list --priority 0                           # 우선순위 필터
tk issue list --project my-app                       # 프로젝트 필터
tk issue list --json                                 # JSON 출력 (자동화용)
tk issue view SDLC-001                               # 상세 보기
tk issue edit SDLC-001 --priority 0                  # 수정
tk issue move SDLC-001 running                       # 상태 변경
tk issue move SDLC-001 paused                        # 일시정지
tk issue move SDLC-001 done                          # 완료
tk issue move SDLC-001 aborted                       # 중단
tk issue comment SDLC-001 "PRD 완료"                 # 코멘트
tk issue delete SDLC-001                             # soft delete
tk issue log SDLC-001                                # 히스토리
```

### project (p) -- 프로젝트 관리

```bash
tk project init                  # 현재 디렉토리를 프로젝트로 등록
tk project init --key APP        # 커스텀 Project Key
tk project list                  # 전체 프로젝트 요약
tk project view                  # 현재 프로젝트 상세
```

### board (b) -- 칸반 뷰

```bash
tk board                         # 현재 프로젝트 칸반
tk board --all                   # 전체 프로젝트 통합 칸반
tk board --status running        # 필터
tk board --tag backend           # 필터
```

### 단축 명령 (jira-cli 패턴)

```bash
tk i c "제목"              # tk issue create
tk i l                     # tk issue list
tk i v SDLC-001            # tk issue view
tk i m SDLC-001 running    # tk issue move
tk b                       # tk board
tk p l                     # tk project list
```

## 상태 전이 규칙

```
backlog --> running --> paused <--> running --> done
backlog --> aborted
paused --> aborted
done/aborted = 종료 상태 (변경 불가)
```

잘못된 전이 시 에러:
```
tk issue move SDLC-001 running
Error: SDLC-001 is 'done'. Cannot move to 'running'. (done is a terminal state)
```

## ID 체계

{PROJECT_KEY}-{NNN}
Project Key: 프로젝트별 설정 (기본값: git repo 이름에서 추출, 대문자)
NNN: 프로젝트 내 순번

## 프로젝트 감지

1. git root에서 프로젝트명 자동 감지
2. 미등록 프로젝트면 "tk project init을 먼저 실행하세요" 안내
3. --project 플래그로 명시적 지정 가능

## factory-run 통합 (v2)

v1: tk CLI 독립 동작
v2: factory-run이 tk를 호출

v1에서 미리 준비:
- --pipeline 옵션 (small/medium/large)
- --json 출력 (프로그래밍 인터페이스)
- handoff 파일 저장 경로 규칙

## Distribution

```bash
bun build ./src/index.ts --compile --outfile tk
cp tk ~/.local/bin/tk
```

## v1 Scope

Core (필수):
- issue: create, list, view, move, delete
- project: init, list, view
- board (현재 프로젝트, --all)

Nice-to-have (여유 있으면):
- issue: edit, comment, log, tags, estimated_hours
- board: --status, --tag 필터
- stats

## Implementation Plan

3. v1 core 명령어 구현 (issue create/list/view/move/delete, project init/list/view, board)

## What I noticed about how you think

- "DB가 있어야 편하지 않을까?"에서 JSON 전제를 깨뜨렸다. 검색이 필요한 데이터는 DB가 맞다.
- "둘 다 쓰도록 하고 싶다"에서 하이브리드가 나왔다. 이분법이 아니라 강점을 합치는 사고.
- "정말 명확한것 같아?"에서 내 설계의 모호함을 짚었다. 다른 사람이 괜찮다고 해도 본인이 납득 안 되면 멈추는 것.
- "잘 모르겠으면 다른 잘 되어있는 것들을 참고하면 좋지!"에서 gh, jira-cli 패턴을 가져오게 됐다. 바퀴를 재발명하지 않는 실용주의.
